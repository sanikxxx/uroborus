# RAG-модуль: гибридный поиск по базе знаний

## Общая архитектура

```
Запрос пользователя
        │
        ▼
   ┌─────────┐
   │ search() │  ← точка входа (agent/modules/rag/search.py)
   └────┬────┘
        │
        ▼
 ┌──────────────┐
 │ VectorStore  │  ← ядро модуля (agent/modules/rag/vector_store.py)
 │              │
 │  .query()    │──► гибридный скоринг ──► top-k документов
 └──────┬───────┘
        │
   ┌────┴────┐
   │         │
   ▼         ▼
Semantic   Keyword
 Score      Boost
 (70%)      (30%)
```

## Компоненты

### 1. База знаний — `knowledge_base.yaml`

YAML-файл с документами. Каждый документ содержит:

```yaml
- id: 23
  title: "Тарифы C2B"
  content: "С 2027 года комиссия для бизнеса..."
  metadata:
  category: "тарифы"
  tags: ["C2B", "комиссия"]
```

### 2. Модель эмбеддингов

Используется `sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2` — мультиязычная модель, поддерживающая русский язык. Преобразует текст в 384-мерный вектор.

### 3. Текст для эмбеддинга — `_doc_text()`

Для каждого документа формируется строка из **всех информативных полей**:

```
{title}. {content}. {category}. {tag1 tag2 tag3}
```

Пример: `"Тарифы C2B. С 2027 года комиссия для бизнеса... тарифы. C2B комиссия"`

Включение `category` и `tags` обогащает эмбеддинг ключевыми словами, которых может не быть в `title`/`content`.

### 4. Кэширование — `index.json`

Эмбеддинги вычисляются один раз и сохраняются в `index.json`. При следующем запуске:

1. Сравнивается `mtime` файла `knowledge_base.yaml` с сохранённым
2. Проверяется совпадение количества эмбеддингов и документов
3. Если всё совпало — загрузка из кэша (мгновенно), иначе — пересчёт

При изменении базы знаний нужно удалить `index.json` или просто изменить `knowledge_base.yaml` — индекс пересоберётся автоматически.

## Гибридный поиск — `query()`

Чисто семантический поиск (cosine similarity) плохо находит документы по прямым ключевым словам. Например, запрос "тарифы цифрового рубля" мог возвращать "Лимиты операций" вместо реальных документов о тарифах.

Решение — **гибридный скоринг** из двух сигналов:

### Семантический скор (вес 0.7)

```python
query_emb = model.encode([text])
semantic_scores = query_emb @ embeddings.T
```

Косинусное сходство между вектором запроса и векторами документов. Ловит смысловую близость, синонимы, перефразировки.

### Keyword boost (вес 0.3)

```python
query_words = set(text.lower().split())
keyword_score = совпавшие_слова / всего_слов_в_запросе
```

Доля слов запроса, которые встречаются в тексте документа (title + content + category + tags). Ловит прямые вхождения ключевых слов.

### Финальный скор

```python
score = 0.7 * semantic + 0.3 * keyword
```

| Вес | Компонент | Что ловит |
|-----|-----------|-----------|
| 0.7 | Semantic  | "как работает цифровая валюта ЦБ" → находит "Цифровой рубль — определение и суть" |
| 0.3 | Keyword   | "тарифы" → бустит документы со словом "тарифы" в title/tags/category |

### Параметр k

По умолчанию `k=5`. При 25 документах в базе это возвращает 20% самых релевантных, что даёт LLM достаточно контекста для ответа.

## Поток данных (end-to-end)

```
1. Пользователь: "какие тарифы действуют?"
2. Агент выбирает навык: rag-answer
3. LLM вызывает инструмент: rag_answer(query="какие тарифы действуют?", k=5)
4. VectorStore.query():
   a. Encode запрос → вектор [384]
   b. Semantic: cosine similarity с 25 документами
   c. Keyword: доля совпавших слов для каждого документа
   d. Финальный скор = 0.7·semantic + 0.3·keyword
   e. Сортировка, top-5
5. LLM получает 5 документов → формирует ответ пользователю
```
